# WorkTwin 飞书多维表格数据模型

> 更新日期：2026-06-20

WorkTwin 使用一个飞书多维表格 Base（`app_token`）承载多个业务子表（`table_id`）。用户认证表已经接入；其余产品表通过统一 Bitable repository 读写。

## 表清单

| 配置键 | 子表名称 | 主键字段 | 写入模式 | 用途 |
| --- | --- | --- | --- | --- |
| `users` | 用户 | 用户ID | 可更新 | 登录身份、密码哈希、角色、账号状态 |
| `wallets` | 钱包 | 钱包ID | 可更新 | CNY/UT 余额和乐观锁版本 |
| `employees` | 数字员工 | 员工ID | 可更新 | 分身身份、简历摘要、定价和状态 |
| `capabilities` | 能力 | 能力ID | 可更新 | MCP Tool、Skill、A2A、Skill Pack 目录 |
| `employeeCapabilities` | 员工能力关联 | 关联ID | 可更新 | 员工与能力多对多关系、启用状态、风险等级 |
| `platformBindings` | 平台绑定 | 绑定ID | 可更新 | OpenClaw/Hermes/Cursor/Claude/Custom 连接状态 |
| `integrationSessions` | 接入会话 | 会话ID | 可更新 | 鉴权、能力发现、组装草稿状态机 |
| `contracts` | 雇佣合同 | 合同ID | 可更新 | 雇佣关系、计费条款和履约汇总 |
| `tasks` | 任务 | 任务ID | 可更新 | 派单、执行、验收与交付结果 |
| `settlements` | 结算流水 | 结算ID | 只增 | CNY/UT 资金记账和幂等结算 |
| `reviews` | 评价 | 评价ID | 只增 | 任务评价和员工信誉证据 |
| `apiKeys` | API密钥 | 密钥ID | 可更新 | Agent API Key 哈希、权限和撤销状态 |
| `webhooks` | Webhook | WebhookID | 可更新 | 回调订阅、状态和签名密钥引用 |
| `auditLogs` | 审计日志 | 审计ID | 只增 | 写操作、权限变化和资金动作审计 |

完整字段定义可通过以下接口读取：

```text
GET /api/data/schema
```

字段定义源码位于 `web/lib/server/feishu-product-schema.ts`。

## 核心关系

```text
用户
 ├─ 1:1 钱包
 ├─ 1:N 数字员工
 ├─ 1:N 雇佣合同（雇主）
 ├─ 1:N API密钥
 └─ 1:N Webhook

数字员工
 ├─ N:M 能力（员工能力关联）
 ├─ 1:N 平台绑定
 ├─ 1:N 雇佣合同
 └─ 1:N 评价

雇佣合同
 ├─ 1:N 任务
 └─ 1:N 结算流水
```

当前代码用稳定业务 ID 文本建立关系，不依赖飞书“关联记录”字段。这样 API 写入更可控，也便于未来迁移 PostgreSQL。

## 关键数据流

### 注册与登录

1. `users` 按小写邮箱查重。
2. 密码使用 scrypt + 独立随机盐，写入密码哈希和密码盐。
3. 登录成功更新最后登录时间。
4. 禁用用户后，现有会话在下一次 `/api/auth/me` 校验时失效。

### 接入并发布数字员工

1. `integrationSessions` 保存连接器状态和草稿。
2. `capabilities` upsert 发现到的能力。
3. `employees` 保存数字员工主体。
4. `employeeCapabilities` 保存多对多能力绑定和风险等级。
5. `platformBindings` 保存平台身份与连接状态；明文 token 不写入表格。
6. `auditLogs` 追加发布操作。

### 雇佣、派单与结算

1. 雇佣生成 `contracts`。
2. 派单生成 `tasks`，同时写幂等键。
3. 完成任务后创建 `settlements` 只增流水。
4. 更新 `wallets` 时使用版本字段做乐观锁检查。
5. 合同汇总字段用于运营展示，结算流水才是财务事实来源。

## 配置表映射

在被 Git 忽略的 `web/config/feishu-auth.json` 中增加：

```json
{
  "app_token": "base_token",
  "table_id": "tbl_users",
  "tables": {
    "users": "tbl_users",
    "wallets": "tbl_wallets",
    "employees": "tbl_employees",
    "capabilities": "tbl_capabilities",
    "employeeCapabilities": "tbl_employee_capabilities",
    "platformBindings": "tbl_platform_bindings",
    "integrationSessions": "tbl_integration_sessions",
    "contracts": "tbl_contracts",
    "tasks": "tbl_tasks",
    "settlements": "tbl_settlements",
    "reviews": "tbl_reviews",
    "apiKeys": "tbl_api_keys",
    "webhooks": "tbl_webhooks",
    "auditLogs": "tbl_audit_logs"
  }
}
```

原有顶层 `table_id` 继续作为 users 表的兼容配置。

## 健康检查

```text
GET /api/auth/health   # 用户表和认证配置
GET /api/data/health   # 14 张产品表及字段结构
GET /api/data/schema   # 期望的数据表定义
```

`/api/data/health` 会返回每张表的状态：

- `ready`：table_id 已配置且字段完整。
- `missing_table`：配置中缺少 table_id。
- `missing_fields`：表存在但字段不完整。
- `error`：权限、token、网络或飞书 API 错误。

## 一致性与限制

飞书多维表格不等价于事务数据库：

- 同一子表写入必须串行，代码已为每张表建立进程内写队列。
- 批量创建单次限制 500 条，代码会拒绝超限请求。
- 邮箱、幂等键和业务 ID 没有数据库级唯一约束，写入前必须查询去重。
- 不支持跨表事务。任务完成、结算、钱包扣减需要补偿逻辑和定时对账。
- 多实例部署时，进程内写队列无法提供全局互斥，需要外部队列或锁服务。

因此多维表格适合早期商业验证、低并发运营和人工管理。进入高并发支付场景后，应以 PostgreSQL/账本服务作为主存储，飞书改为运营镜像。

## 安全规则

- App Secret、tenant token、原始 API Key、连接器 token 不得进入业务表。
- `apiKeys` 只保存前缀和哈希。
- `platformBindings` 与 `webhooks` 只保存服务端 Secret/Vault 引用。
- `settlements`、`reviews`、`auditLogs` 禁止覆盖历史记录。
- 示例配置不得包含真实凭证；真实配置文件必须保持 Git 忽略。
