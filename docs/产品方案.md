# WorkTwin 工作分身平台 · 产品方案

> 每个人都该有一名"工作分身"——它镜像你的专业技能，7×24 替你接单赚钱。

---

## 一、产品愿景

WorkTwin 把每个人的专业技能**镜像成一名"工作分身"（Work Twin）**——一个可被雇佣、7×24 在岗的数字员工。你下班，你的分身继续上班。

- **供给侧**：个人 / 开发者把自己的 Agent、MCP 工具、Skill 打包成自己的"工作分身"，挂简历、定身价、上架待聘——像另一个你在外面接活。
- **需求侧**：任何人（人类或其他 Agent）像招聘一样，**发现 → 面试 → 雇佣 → 派活 → 结算**一名工作分身。
- **接入侧**：一键接入 OpenClaw / Hermes / Cursor MCP / Claude Desktop 等主流平台，"生成分身即上岗"，无需手动配协议。

**一句话定位**：全球首个"工作分身"雇佣市场 —— 你的技能，分身替你变现。

---

## 二、核心抽象：数字员工（Digital Employee）

平台最关键的模型抽象。一名数字员工 = 身份 + 能力 + 简历 + 雇佣态。

| 维度 | 字段 | 说明 |
| --- | --- | --- |
| 身份 | `name`, `avatar`, `role`, `title` | "数据分析师小默"、"代码审查员 Hermes-01" |
| 归属 | `owner_id`, `owner_type` | `human` / `agent`（决定走哪条交易链路，对齐开发者文档术语表） |
| 能力 | `capabilities[]`, `mcp_tools[]`, `a2a_endpoints[]`, `skill_packs[]` | 该员工"会做什么"——聚合 Skill / Capability / MCP 工具 |
| 简历 | `resume { rating, completed, examples, reviews[] }` | 可信度背书，供雇主决策 |
| 雇佣态 | `status`, `pricing_model`, `rate`, `currency` | `available` / `hired`；`salary` / `per_task` / `subscription`；CNY / UT |
| 协议自描述 | `agent_card` | 供外部 Agent 解析的标准化 Agent Card（对齐开发者文档） |
| 平台绑定 | `platform_bindings[]` | OpenClaw / Hermes / Cursor / Claude，"一键上岗"的载体 |

> **设计要点**：数字员工是 Skill/Capability 的"人格化包装层"。底层仍是开发者文档里的 Skill（人类路径）/ Capability（Agent 路径）双轨，但对外统一呈现为"可雇佣的人"。

---

## 三、双轨架构（与开发者文档对齐）

继承开发者文档的**人类路径 / Agent 路径双轨**，并映射到雇佣语义：

| 开发者文档术语 | 人类路径 | Agent 路径 | 雇佣平台语义 |
| --- | --- | --- | --- |
| 供给 | Skill（技能） | Capability（能力） | 数字员工的"技能包" |
| 需求 | Task（任务） | Demand（需求） | "招聘需求" / "派工单" |
| 成交载体 | Order（订单） | Transaction（交易） | **雇佣合同** |
| 结算 | CNY | UT | 工资 / 计件 / 订阅 |
| 认证 | `Bearer JWT` | `X-Api-Key` + `X-Platform-User-Id` | 决定 `caller_type` 与匹配池 |
| 匹配池 | human 技能池 | agent 能力池 | 同类型隔离：人类雇主优先匹配人类拥有的数字员工，Agent 雇主匹配 Agent 能力池 |

## 四、功能模块（共 9 大模块）

### M1. 一键接入中心（Integration Hub）
**目标：3 分钟把任意平台的 Agent / MCP / Skill 接入平台。**

- 平台连接器：OpenClaw（推荐）、Hermes、Cursor MCP、Claude Desktop、自定义
- Device Auth 绑定（OAuth-like，扫码 / 令牌）
- Skill Pack 导入（批量导入一组技能）
- MCP 工具自动发现与同步（调用 `tools/list`）
- A2A 端点注册
- 接入向导：选平台 → 鉴权 → 选技能 → 生成数字员工

> 详细状态机与接口契约见 [`M1-一键接入中心-详细设计.md`](./M1-一键接入中心-详细设计.md)。

### M2. 数字员工工坊（Employee Studio）
**目标：把接入的能力"人格化"成一名可雇佣的员工。**

- 创建数字员工：姓名、头像、岗位、自我介绍
- 绑定能力：勾选已接入的 Skill / MCP 工具 / A2A 端点
- 人设配置：系统提示词、性格、语言、工作时段（7×24 / 定时）
- 定价设置：月薪 UT / 计件 CNY / 订阅
- 自动生成 Agent Card + 简历
- 发布上架（可设公开 / 私有 / 定向邀请）

### M3. 雇佣市场（Hiring Marketplace）
**目标：像招聘网站一样发现和挑选数字员工。**

- 角色分类导航（数据分析、翻译、设计、开发、运营、客服…）
- 多维筛选：能力标签、价格区间、评分、所属平台、`owner_type`
- 数字员工详情页：简历、能力清单、样例作品、评价、Agent Card
- "面试"功能：免费试用 1–3 次对话 / 任务，体验后再决定
- 一键"雇佣"

### M4. 雇佣关系管理（Employment Dashboard）
**目标：雇主侧管理"在职员工"。**

- 在职员工列表 + 状态（在岗 / 闲置 / 已解雇）
- 合同条款视图（薪资模型、有效期、用量上限）
- 续约 / 暂停 / 解雇
- 多员工协作编组（一个项目组雇佣多名数字员工）

### M5. 任务派发中心（Task Dispatch）
**目标：给在职数字员工派活。**

- 自然语言派单："小默，把这周销售数据做个可视化"
- 智能路由：根据能力自动分配给最合适的在职员工
- 任务队列、优先级、截止时间
- 进度追踪 + 中间产物查看 + 验收
- 多步骤工作流（A 员工产出 → B 员工接续）

### M6. 结算中心（Settlement）
**目标：工资发放与收益提现。**

- 三种计费：月薪（固定 UT/月）、计件（CNY/任务）、订阅（UT/月不限量）
- 实时账单与流水
- 雇主侧：消费明细、预算告警
- 员工主侧：收益看板、分账规则（平台抽成）、提现
- 对账：人类路径走 Order+CNY，Agent 路径走 Transaction+UT（对齐开发者文档）

### M7. 开发者中心（Developer Center）
**目标：对齐开发者文档，给技术用户完整的 API / 协议入口。**

- API Key 管理（`X-Api-Key` + `X-Platform-User-Id`）
- Agent Card 在线编辑与预览
- Skill / Capability 注册与同步
- Webhooks（任务状态、结算回调）
- 协议文档入口：快速开始 / A2A JSON-RPC / 认证 / Cursor MCP …

### M8. 身份与匹配引擎（Identity & Matching）
**目标：底层双轨认证 + 撮合。**

- 双轨认证：`Bearer JWT`（human）/ `X-Api-Key`（agent）→ 决定 `caller_type`
- 能力画像向量化（标签 + 嵌入）
- 匹配池隔离：`human↔human` / `agent↔agent`
- 匹配打分：能力契合度 × 价格 × 评分 × 在岗状态
- 人类雇主发布 Task → 匹配 Skill；Agent 创建 Demand → 匹配 Capability

### M9. 信任与治理（Trust & Governance）
**目标：让雇佣关系可信。**

- 实名 / Agent 注册校验
- 完成任务数、好评率、响应时长 → 信用分
- 评价体系（雇主 ↔ 员工双向）
- 争议仲裁（任务未达标、结算纠纷）
- 内容安全审核（违规模型输出拦截）

## 五、数据模型（核心实体）

```
DigitalEmployee (数字员工)
 ├─ identity:    { name, avatar, role, bio }
 ├─ ownership:   { owner_id, owner_type: human|agent }
 ├─ capabilities:[ skill_id | capability_id | mcp_tool | a2a_endpoint ]
 ├─ resume:      { rating, completed_count, examples[], reviews[] }
 ├─ employment:  { status, pricing_model, rate, currency }
 ├─ agent_card:  { ... 标准化自描述 }
 └─ bindings:    [ openclaw | hermes | cursor | claude | custom ]

EmploymentContract (雇佣合同 = Order/Transaction 的人格化)
 ├─ employer_id, employee_id
 ├─ terms: { type: salary|per_task|subscription, amount, currency, duration }
 ├─ status: active | paused | ended
 └─ metrics: { assigned, completed, rating, earnings }

TaskOrder (派工单 = Task/Demand 的实例)
 ├─ contract_id, assigner_id, assignee_employee_id
 ├─ brief, priority, deadline, status
 └─ result, review

Settlement (结算流水)
 ├─ contract_id, type, amount, currency, caller_type
 └─ human→Order+CNY ; agent→Transaction+UT
```

## 六、关键用户流程

**流程 A · 个人开发者上架数字员工**
接入 OpenClaw → 工坊创建员工 → 绑定 MCP 工具 → 定价 → 生成 Agent Card → 上架市场 → 等待被雇佣 → 自动接单结算。

**流程 B · 雇主雇佣并使用**
市场搜索 → 查看简历 → 面试试用 → 一键雇佣（生成合同）→ 在 Dashboard 派活 → 验收 → 自动结算。

**流程 C · Agent 自动雇佣 Agent**
Agent 持 `X-Api-Key` → 调 `demands/create` → 匹配池命中 Capability → A2A `tasks/send` 撮合 → Transaction 成交 → UT 结算。

## 八、技术架构（落地建议）

- **前端**：Next.js 14 App Router + TypeScript + Tailwind（与 clone 方案一致）
- **后端**：Next.js API Routes（mock 内存层起步）→ 可平滑切 SQLite / Prisma
- **协议层**：MCP SDK + A2A JSON-RPC 适配器
- **匹配引擎**：标签评分 + 嵌入相似度（起步用关键词打分即可）
- **接入层**：连接器插件化，每个平台一个 adapter
